## Цель

Зафиксировать технический подход к управлению характеристиками товаров в админке и API для доменов каталога/категорий.

## Scope

- Динамические характеристики товара на уровне подкатегории.
- Версионирование схем характеристик.
- Валидация значений характеристик на стороне API.

## Принятые решения

- Набор характеристик задается **независимо для каждой подкатегории**.
- Используется **версионирование схемы** характеристик.
- Изменения схемы не ломают ранее сохраненные товары.

## Модель данных (концепт)

### CategoryAttributeSchema

- `id`
- `subcategoryId`
- `version`
- `status` (`draft` | `published`)
- `fields[]`
- `createdAt`
- `updatedAt`

### CategoryAttributeField

- `key` (уникальный в рамках схемы)
- `label`
- `type` (`text` | `number` | `boolean` | `select` | `multiselect`)
- `required`
- `options[]` (для `select`/`multiselect`)
- `sortOrder`
- `validation` (min/max/regex при необходимости)

### Product (часть по характеристикам)

- `subcategoryId`
- `schemaVersion`
- `attributesSnapshot` (ключ-значение, валидируемое по версии схемы)

## Поведение в админке

1. Менеджер открывает подкатегорию и вкладку "Характеристики".
2. Создает или редактирует `draft` схемы.
3. Публикует новую версию схемы.
4. При создании/редактировании товара форма характеристик строится автоматически по активной версии.

## Поведение в API

- API валидирует `attributesSnapshot` по активной/указанной версии схемы.
- Неизвестные ключи характеристик отклоняются.
- Для `select`/`multiselect` принимаются только значения из `options`.
- Обязательные поля проверяются на заполнение.

## Версионирование

- Новая публикация создает следующую версию.
- Старые товары остаются на старой версии (историческая консистентность).
- Новые товары и обновления используют актуальную опубликованную версию.

## Базовые endpoints (MVP)

- `GET /api/categories/:id/attributes/schema`
- `POST /api/categories/:id/attributes/schema/draft`
- `POST /api/categories/:id/attributes/schema/publish`
- `POST /api/catalog`
- `PATCH /api/catalog/:id`

## Ограничения MVP

- Без наследования характеристик от родительской категории.
- Без автоматической миграции старых товаров на новую схему.
- Без продвинутого конструктора форм (UI строится из базовых типов).

## Критерии готовности

- Для каждой подкатегории существует опубликованная схема характеристик с версией.
- Создание/редактирование товара валидируется по версии схемы без ручных обходов.
- Старые товары остаются консистентными после публикации новой версии схемы.

## Out of scope

- Универсальный low-code form builder.
- Глубокая иерархия наследования схем между категориями.
- Автоматическая массовая миграция атрибутов исторических товаров.
