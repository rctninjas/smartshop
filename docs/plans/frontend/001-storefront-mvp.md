## Цель

Определить минимально достаточный scope витрины (`storefront`) для запуска продаж: просмотр каталога, карточка товара, добавление в корзину и базовый сценарий оформления заказа.

## Источник требований

- Основной источник: [docs/initial-requirements.txt](../../initial-requirements.txt), раздел `Процесс заказа`.

## Scope MVP

В MVP витрины входят:

- Список товаров (каталог).
- Карточка товара.
- Галерея изображений товара.
- Отображение названия, категории, цены и скидки (если есть).
- Кнопка `Добавить в корзину`.
- Страница корзины.
- Флоу оформления заказа:
  - проверка корзины;
  - вход/регистрация перед оформлением (gate);
  - ввод данных доставки;
  - выбор доставки (`СДЭК` / `Почта России`);
  - выбор способа оплаты;
  - обработка результата оплаты;
  - создание заказа и подтверждение.

## Пользовательские сценарии

### 1) Просмотр и выбор товара

- Пользователь открывает витрину и переходит в каталог.
- Просматривает карточки товаров.
- Переходит на страницу товара.
- Смотрит изображения, цену/скидку, категорию.
- Принимает решение:
  - не подходит -> назад в каталог;
  - подходит -> добавляет в корзину.

### 2) Добавление в корзину

- Пользователь нажимает `Добавить в корзину`.
- Система предлагает:
  - продолжить покупки;
  - перейти в корзину.

### 3) Корзина и начало checkout

- Пользователь проверяет состав корзины.
- Система проверяет авторизацию:
  - если пользователь не авторизован -> вход/регистрация;
  - если авторизован -> переход к доставке.

## Регистрация и авторизация клиента (MVP)

### Обязательные поля регистрации

- `fullName` — имя и фамилия в одном поле.
- `email`
- `phone`
- `password`
- `passwordConfirm`
- `consent` — чекбокс согласия с политикой/офертой.

### Варианты входа

- `email + password`
- `email + oneTimeCode`

Правило: если введен пароль и он валиден, одноразовый код не требуется.

### 4) Доставка

- Пользователь вводит данные доставки.
- Выбирает способ доставки:
  - `СДЭК`;
  - `Почта России`.

### 5) Оплата и создание заказа

- Пользователь выбирает способ оплаты.
- Система проверяет результат оплаты:
  - неуспешно -> возврат к выбору способа оплаты;
  - успешно -> заказ создается.
- Пользователь получает подтверждение (`email/SMS`).

### 6) Завершение заказа

- Заказ передается на склад и в службу доставки.
- После получения клиентом заказ считается завершенным.

## Экранные требования MVP

### Каталог

- Сетка карточек товаров.
- На карточке товара обязательно:
  - фото (preview),
  - название,
  - категория,
  - цена,
  - скидка (если есть),
  - кнопка `Добавить в корзину`.

### Карточка товара

- Галерея фото (минимум 1 изображение).
- Название, категория, цена, скидка.
- Кнопка `Добавить в корзину`.

### Корзина

- Список добавленных товаров.
- Подытог заказа.
- Переход к оформлению.

### Checkout

- Шаг авторизации (если нужно).
- Шаг доставки.
- Шаг оплаты.
- Шаг подтверждения оформления.

## UI/UX требования

- Все тексты пользовательского интерфейса витрины — на русском языке.
- Корректная работа на desktop и mobile.
- Явные состояния:
  - `loading`,
  - `empty`,
  - `error`.
- Для цены и скидки:
  - если скидка есть, отображаются обе цены (`price` и `sale`);
  - если скидки нет, отображается только `price`.

## Минимальные API требования для MVP

- Получение списка опубликованных товаров для каталога.
- Получение детальной карточки товара по `slug/id`.
- Создание заказа из корзины.
- Проверка/обработка статуса оплаты в рамках checkout flow.

## Критерии приемки

- Пользователь может открыть каталог и увидеть карточки товаров с обязательными полями.
- Пользователь может открыть карточку товара и просмотреть галерею фото.
- Пользователь может добавить товар в корзину и перейти к оформлению.
- Checkout повторяет согласованный порядок из `Процесс заказа`.
- При успешной оплате создается заказ и показывается подтверждение.
- При неуспешной оплате пользователь возвращается к выбору способа оплаты.

## Out of scope для MVP

- Wishlist, сравнение товаров, отзывы, персональные рекомендации.
- Сложные промо-механики, купоны, мультивалютность.
- Продвинутые сценарии личного кабинета.
